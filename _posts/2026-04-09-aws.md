# S3
## 核心概念
S3 是**对象存储（Object Storage）**，而不是分层文件系统。

- **Bucket（存储桶）**：存储对象的容器。**Bucket 名称在全球所有 AWS 账户中必须是唯一的**。
- **Object（对象）**：你上传的文件。每个对象由两部分组成：
	- **Data（数据本身）**
	- **Metadata（元数据）**：键值对（Key-Value pairs），记录文件的类型、大小、创建时间或自定义标签。
- **Key（键）**：文件的“绝对路径”（例如 `raw-data/2026/06/09/log.json`）。在底层，S3 是扁平的键值结构，UI 里的“文件夹”其实只是由分隔符（`/`）产生的前缀（Prefix）视觉效果。
- **无限容量与单文件上限：** 你可以在一个 S3 桶（Bucket）里塞进任意 PB 级别的数据，它会自动横向扩展。但注意：**单个 Object（文件）的最大上限是 5 TB**。如果超过 5 TB，必须使用分段上传（Multipart Upload）。


- **Durability（持久性 - 11个9）**：`99.999999999%`。这意味着如果你在 S3 中存储 10,000 个对象，平均每 1000 万年才会丢失一个文件。S3 默认会把你的文件自动复制到同一 Region 内的至少 3 个物理可用区（AZ）。
- **Availability（可用性 - 4个9）**：`99.99%`。指服务在一年中可以正常访问的时间比例。S3 standard has 99.99% availability = not available 53 min a year.

## S3 versioning (版本控制)

Amazon S3 的 **Versioning（版本控制）** 是一个在**存储桶（Bucket）级别**开启的功能。简单来说，它就像是为你的 S3 存储桶配备了一个“时光机”**和**“回收站”。

在理解 Versioning 时，最核心的观念是：**开启版本控制后，S3 桶中的文件永远不会被真正“覆盖”或“直接删除”，而是会保留所有的历史痕迹。**

### 1\. 核心工作原理

#### A. 文件的“覆盖”操作 

- **未开启 Versioning**：如果你上传了一个名为 `config.json` 的文件，接着又上传了一个同名的 `config.json`，旧文件的内容会被新内容**直接覆盖**，旧数据永久丢失。
- **开启 Versioning**：每次你上传同名文件时，S3 不会覆盖原文件，而是会为这个文件生成一个唯一的 **Version ID**（版本 ID）。
	- 最新的那次上传会被标记为 **Current Version（当前版本）**。
		- 之前上传的同名文件会变成 **Noncurrent Version（历史版本）**。
		- 当你通过代码或 API 请求 `config.json` 且不指定版本号时，S3 默认永远返回 **Current Version**。

#### B. 文件的“删除”操作 


在开启版本控制的桶里，当你点击“删除”一个文件时：

1. S3 并没有真正抹去任何数据。
2. S3 会在最上面放置一个特殊的对象，叫做 **Delete Marker（删除标记）**。
3. 这个 **Delete Marker** 顶替它成为了 **Current Version（当前版本）**。
4. 此时你在 AWS 控制台（默认视图）或通过标准 API 访问该文件，S3 会返回 `404 Not Found`，假装文件已经被删除了。

> 💡 **如何“复活”被删的文件？** 你只需要在控制台打开“显示版本（Show versions）”开关，找到那个处于最顶层的 `Delete Marker` 并把它**彻底删除**。一旦删除了删除标记，前一个历史版本就会重新变成“当前版本”，文件完美复活。

---
### 2\. 状态切换与收费逻辑

关于 Versioning 的状态，你需要知道以下三点：

1. **状态不可逆（只能开启或暂停）**： 一个 S3 桶的 Versioning 状态只有三种：
	- 未开启（Unversioned）
	- 已开启（Enabled） 
	- 已暂停（Suspended）。一旦开启过，你就再也无法把它变回“未开启”状态，只能选择“暂停”。
2. **暂停后的行为**： 如果暂停了版本控制，新上传的同名文件会覆盖 `Version ID` 为 `null` 的那个版本，但**之前已经存在的历史版本依然会安全地躺在里面**，不会消失。
3. **空间与计费**： **每一个历史版本都会按照实际大小完整计费！** 比如一个 10GB 的文件，你修改并上传了 5 次，在开启版本控制的情况下，S3 里会堆积 5 个版本，总共计算 **50GB** 的存储费用。这也是为什么不能盲目开启版本控制、或者==开启后必须配合生命周期管理的原因==。

### 3\. 彻底删除（Permanent Delete）怎么做？

如果你因为合规、清空空间等原因，必须**彻底、物理删除**一个文件，你需要怎么做？

- 你必须在删除请求中**明确指定 `Version ID`**。
- 只有显式指定了版本 ID 的删除操作，S3 才会把该版本从硬盘上永久抹去。
- 如果你要清空桶，必须写脚本或配置规则，把所有的历史版本和所有的 `Delete Marker` 全部彻底删除。

### 4\. 数据工程中的生产实践

在实际的数据管道（ETL）和数据湖架构中，Versioning 通常与以下特性结合使用：

- **防止误操作/勒索病毒（MFA Delete）**： 可以配置为：如果想要“彻底删除”某个历史版本或关闭版本控制，必须提供根账号的 **MFA（多因素验证码）**。这样即使生产环境的 IAM 密钥泄露，黑客也无法恶意清空你的历史数据资产。
- **配合生命周期策略（Lifecycle Rules）进行成本控制**： 因为历史版本很费钱，通常会配置生命周期规则：
	> “当前版本（Current）正常保留；一旦变成历史版本（Noncurrent），30天后自动转入便宜的 `Glacier` 归档存储，90天后彻底物理删除。”
- **数据审计与回滚**： 在机器学习训练或数据分析中，如果你发现今天的模型训练结果异常，怀疑是输入的数据源被污染了，你可以通过 Versioning 轻松追溯并读取到昨天、甚至前天那个时间节点的历史数据快照。

## S3 replication
**Amazon S3 Replication** 是 AWS 提供的一项功能，用于在不同的 Amazon S3 存储桶（Buckets）之间实现**自动、异步的跨区域或同区域对象复制**。

只要你在源存储桶上配置了复制规则，任何**新上传**的对象、对象更新或标签更改，都会被 S3 自动且在后台复制到目标存储桶中。

### 核心工作原理

S3 Replication 核心要求源存储桶和目标存储桶都**必须开启版本控制（Versioning）**。它主要分为两种类型：

- **CRR (Cross-Region Replication，跨区域复制)：** 源存储桶和目标存储桶位于**不同的 AWS 区域**（例如：从美西旧金山复制到美东弗吉尼亚）。
- **SRR (Same-Region Replication，同区域复制)：** 源存储桶和目标存储桶位于**相同的 AWS 区域**。

### 典型使用场景

以下是几种最常见的工业级应用场景：

#### 1\. 合规性与容灾备份 (DR)

- **多区域容灾：** 为了防止某个 AWS 区域发生罕见的重大灾难或基础设施故障，使用 **CRR** 将核心数据实时备份到另一个地理位置相隔很远的区域，确保业务具备超强的容灾恢复能力（DR）。
- **主备切换：** 当主业务区不可用时，应用可以迅速切换到备用区域的 S3 存储桶读取数据。

#### 2\. 降低全球用户的访问延迟 (Latency Optimization)

- 如果你的公司总部在美国，但亚洲有大量的用户。直接让亚洲用户跨洋访问美国的 S3 存储桶速度会很慢。
- 通过 **CRR** 将数据复制到东京或新加坡的存储桶，让亚洲的用户直接访问本地的存储桶，大幅提升下载和加载速度。

#### 3\. 数据主权与合规性要求 (Compliance)

- 某些国家的法律法规（如 GDPR 或特定金融监管）严格要求某些特定数据必须存储在境内。
- 你可以通过复制规则，将特定前缀（Prefix）或带有特定标签（Tags）的数据自动路由并存储到符合合规要求的特定区域存储桶中。

#### 4\. 生产与测试环境的数据隔离 (Environment Isolation)

- 在日常开发中，可能需要使用生产环境的真实数据来做大数据分析、机器学习模型训练或严格的测试。
- 为了防止分析或测试脚本不小心误删、污染生产数据，可以使用 **SRR** 或 **CRR** 将生产存储桶的数据自动同步到一个完全独立的“分析/测试存储桶”中，实现安全隔离。

#### 5\. 跨账号所有权转移 (Cross-Account Ownership)

- 在企业多账号管理架构下（比如一个专门的 Log 账号，一个 Production 账号），可能需要将数据从一个 AWS 账号迁移或分享给另一个 AWS 账号。
- S3 Replication 支持跨账号复制，并且可以在复制的同时**自动更改目标对象的所有权（Ownership）**，方便接收方账号直接控制和管理这些文件。

### 关键技术特性（常考）

- **异步复制：** 复制不是瞬间完成的（非同步写入）。虽然大部分对象在几分钟内就能完成复制，但标准服务并不打包票具体的完成时间。如果对时间有严格要求，可以开启 **S3 Replication Time Control (RTC)**，它承诺 99.99%的对象会在 15 分钟内复制完成。
- **非追溯性（历史数据默认不复制）：** 在你开启复制规则的那一刻起，只有**新上传**或**被修改**的对象会被自动复制。如果你需要复制开启前存储桶里的“存量历史数据”，需要配合使用 **S3 Batch Operations（批处理操作）** 来完成。
- **删除行为（Delete Markers）：** 默认情况下，如果你在源存储桶删除了一个文件（产生了一个删除标记），这个删除标记**不会**复制到目标存储桶，这样可以防止源存储桶被恶意投毒或误删时，备份数据也被瞬间毁掉。当然，这个行为可以通过配置来调整。

## S3 Event notification

**Amazon S3 Event Notifications（事件通知）** 是 S3 非常强大的一个“主动通知”机制。

如果把 S3 比作一个**数据仓库**，通常情况下它是**被动的**（你用代码去查，它才给你数据）。但开启了 Event Notifications 后，它就变成了一个**自带全天候保安的智能仓库**——只要仓库里发生任何风吹草动（比如新放进来一个文件，或者删除了一个文件），保安就会立刻拉响警报，通知后台的自动化部队（如 Lambda、消息队列）开始干活。

这是构建事件驱动架构（**Event-Driven Architecture**）**和**自动化数据管道（**ETL**）的核心纽带。

### 1\. 三个核心概念：谁触发了什么，通知了谁？

理解 S3 Notification 只需要理清一个三部曲：**Event Source（事件源） →Event Type（事件类型） → Event Destination（接收目的地）**。

#### 概念一：Event Source（事件源与过滤器）

这就是你的 S3 Bucket。但你通常不想让桶里“任何文件的任何变动”都引发警报。所以 S3 允许你使用 **Prefix（前缀）** 和 **Suffix（后缀）** 进行精准过滤。

- **场景**：你只想在有人把图片上传到 `images/` 文件夹时触发通知。
- **配置**：Prefix 设为 `images/`，Suffix 设为 `.jpg` 或 `.png`。这样，上传 `data/config.json` 时系统就会保持安静。

#### 概念二：Event Type（事件类型）

到底发生了什么事？S3 支持非常细粒度的事件订阅：

- `s3:ObjectCreated:*`：只要**创建/上传**对象就触发（包括 `Put`、`Post`、`Copy`、以及大文件分片上传完成 `CompleteMultipartUpload`）。
- `s3:ObjectRemoved:*`：文件被**删除**或放置了删除标记时触发。
- `s3:ObjectRestore:*`：当从 Glacier 归档中**恢复**文件时触发。
- `s3:LifecycleExpiration:*`：当文件因为生命周期策略到期被**自动彻底删除**时触发。

#### 概念三：Event Destination（目的地 / 谁来接单）

警报拉响了，通知发给谁？S3 目前原生且最主流支持的**三大目的地**（俗称 S3 通知的“三驾马车”）：

1. **AWS Lambda（最常用）**：直接触发一段数据处理函数。
2. **Amazon SQS（简单队列服务）**：把事件塞进一个消息队列里，让后方的分布式消费者慢慢消费（适合高并发、抗削峰场景）。[^1]
3. **Amazon SNS（简单通知服务）**：把事件广播出去，可以发邮件、发短信，或者同时触发多个不同的下游组件。

### 2\. 经典数据工程场景：带你脑补它怎么玩

让我们用一个最经典的“用户上传头像自动裁剪”**或**“日志落地自动清洗”的场景来体会它：

1. **动作**：前端用户在 App 上修改了头像，一张 `user_123.png` 被成功上传到了 S3 桶的 `raw-avatars/` 路径下。
2. **触发**：S3 检测到满足条件（前缀 `raw-avatars/`，事件 `ObjectCreated:Put`），立刻把这次上传事件打包成一段 **JSON 格式的通知消息**（里面包含：谁在几点几分，往哪个桶的哪个路径，传了一个多大的文件）。
3. **响应**：S3 异步将这个 JSON 发送给 **AWS Lambda**。
4. **处理**：Lambda 函数被瞬间唤醒，从 JSON 中拿到路径，去 S3 读出这张 `user_123.png`，在内存里把它裁剪成 `50x50` 的缩略图，最后把缩略图写回 S3 的另一个 `thumbnail-avatars/` 路径下。

整个过程**不需要服务器 24 小时死循环去轮询（Poll）** S3 看看有没有新文件。没有文件时，不花一分钱；一旦有文件，毫秒级响应。

### 3\. 必须知道的“避坑指南”（架构师必修）

在设计 S3 Notification 时，有两个非常致命的隐形陷阱，哪怕是资深开发者也容易踩坑：

#### 陷阱一：死循环（Infinite Loop）—— 账单爆炸机

- **错误架构**：你配置了一个通知：“只要桶里有**任何**新文件创建，就触发 Lambda”。而你的 Lambda 代码逻辑是：“读取这个文件，处理一下，然后**写回同一个桶的另一个文件夹里**”。
- **惨剧发生**：Lambda 写回新文件的动作，再次触发了 `ObjectCreated` 事件 →又唤醒了 Lambda →Lambda 又写了一个新文件 →又触发通知……
- **后果**：Lambda 和 S3 开始疯狂自我套娃，几分钟内可能产生几十万次调用，直到你的 AWS 账户额度被扣光。
- **解法**：**必须精准配置 Prefix/Suffix**。让 Lambda 监听 `input/` 前缀，并把结果写到 `output/` 前缀，两者绝对不要重叠。

#### 陷阱二：跨账号通知（Cross-Account）的限制

S3 的 Event Notification **默认只能发送给同一个 AWS 账号下的 SNS、SQS 或 Lambda**。 如果你遇到了极其复杂的企业架构，需要把 A 账号的 S3 变动通知给 B 账号的组件，普通的 S3 Notification 就做不到了。这时候你需要求助于 **Amazon EventBridge**（把 S3 事件发送到全局事件总线，再路由给跨账号的目的地）。

### 4\. 实操

#### 第一步：创建“蓄水池” —— SQS 队列

1. 登录 AWS 控制台，在顶部搜索栏输入 **SQS**，点击进入 **Simple Queue Service**。
2. 点击右上角的 **创建队列 (Create queue)**。
3. **类型 (Type)**：选择默认的 **标准 (Standard)**。
4. **名称 (Name)**：输入 `s3-notification-queue`。
5. 向下滚动，其余所有参数（包括默认配置）全部保持不变，直接点击最下方的 **创建队列 (Create queue)**。

🚨 核心关键：修改 SQS 的 Access Policy（允许 S3 往里扔消息）

默认情况下，SQS 拒绝接收任何外部服务发来的消息。我们需要给 S3 开绿灯：

1. 在刚刚创建好的队列详情页中，切换到 **访问策略 (Access policy)** 标签页。
2. 点击右上角的 **编辑 (Edit)**。
3. 替换或在 JSON 中添加以下策略（记得把 `你的账号ID` 和 `你的桶名` 替换掉）：
JSON

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ToPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "SQS:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:你的账号ID:s3-notification-queue",
      "Condition": {
        "ArnLike": {
          "aws:SourceArn": "arn:aws:s3:::你的桶名"
        }
      }
    }
  ]
}
```
4. 点击 **保存 (Save)**。
5. **复制保存该队列的 ARN**（在详情页顶部可以找到，格式类似于 `arn:aws:sqs:us-east-1:123456789012:s3-notification-queue`），下一步会用到。

### 第二步：在 S3 中配置 Event Notification

现在我们回到上一节课创建的 S3 桶中（如果没有，可以新建一个普通的桶）。

1. 进入你的 **S3 存储桶详情页**。
2. 点击顶部的 **属性 (Properties)** 标签页。
3. 向下滚动，找到 **事件通知 (Event notifications)** 这一栏。
4. 点击 **创建事件通知 (Create event notification)**。

#### 填写通知规则：

- **事件名称 (Event name)**：输入 `incoming-logs-trigger`。
- **前缀 (Prefix)**（可选）：输入 `logs/`。*（这意味着只有上传到 logs/ 文件夹下的文件才会触发通知，防止死循环或误触发）*。
- **后缀 (Suffix)**（可选）：输入 `.json`。
- **事件类型 (Event types)**：向下滚动，在“对象创建”下勾选 **All object create events**（或者只勾选 `s3:ObjectCreated:Put`）。
- **目标 (Destination)**：拉到最底部，选择 **SQS 队列 (SQS queue)**。
- **SQS 队列**：选择指定 SQS 队列 
- 点击 **保存更改 (Save changes)**。

> 💡 **成功标志**：如果保存成功，说明你的 S3 和 SQS 已经顺利接通！如果报错，99% 的原因是因为第一步中 SQS 的 Access Policy 里的 ARN 或桶名没有写对，S3 被 SQS 拒之门外了。

### 第三步：见证奇迹的时刻（测试触发）

我们现在来模拟真实的数据管道流转。

#### 1\. 往 S3 扔一个文件

1. 回到 S3 桶的 **对象 (Objects)** 标签页。
2. 点击 **创建文件夹 (Create folder)**，创建一个名为 `logs` 的文件夹（因为我们配置了前缀过滤）。
3. 进入 `logs/` 文件夹，点击 **上传 (Upload)**，随便传一个 `.json` 后缀的文件（例如 `test_log.json`）。

#### 2\. 去 SQS 拦截这封“匿名举报信”

1. 切换回你的 **SQS 控制台**，点击进入你的 `s3-notification-queue`。
2. 在右上角点击 **发送和接收消息 (Send and receive messages)**。
3. 向下滚动，点击右侧的 **轮询消息 (Poll for messages)** 按钮。
4. **你会发现队列里赫然出现了一条新消息！**
5. 点击这条消息的 ID 打开它，切换到 **Body** 视图。你会看到一大串 JSON 文本，仔细阅读它，你会发现里面清晰地记录着：
	- `awsRegion`: 哪个区域
		- `bucket -> name`: 发生变动的桶名
		- `object -> key`: 触发事件的文件路径（`logs/test_log.json`）
		- `object -> size`: 文件的大小

后端的消费程序（如 Python 脚本）就是通过拿取这条消息，解析出其中的 `key`，然后精准去 S3 对应位置下载该文件进行清洗的。

## S3 Security：安全与权限控制

S3 早期因为用户配置不当经常发生“数据泄露”新闻，现在 AWS 默认关闭了所有公开访问。安全控制主要通过以下手段：

- **Block Public Access（阻止公共访问）**：最高级别的安全开关，开启后任何人都无法通过公网将桶公开。
- **Bucket Policy（存储桶策略）**：基于资源的 JSON 策略，控制**谁**（哪个 AWS 账户或 IAM 角色）可以对这个存储桶执行什么操作。
- **IAM Policy（身份策略）**：挂载在用户或组件（如 Lambda、EC2）上的策略，规定它们能访问哪些 S3 资源。
- **数据加密（Encryption）**：
	- **静态加密（Encryption at Rest）**：现在所有上传到 S3 的数据默认通过 SSE-S3 进行服务器端加密。也可以使用 AWS KMS 密钥（SSE-KMS）进行更高级的管理。
		- **传输中加密（Encryption in Transit）**：强制使用 HTTPS（TLS）来传输数据。


AWS S3 的安全与权限控制（Security & Access Management）是 AWS 生态中最重要、也最容易出错的板块。在数据工程和架构设计中，确保数据“既能被授权的组件高效读取，又绝不向公网泄露”是核心任务。

我们可以将 S3 的安全体系拆解为三大支柱：**访问控制（谁能进）**、**网络防线（怎么进）** 和 **数据加密（丢了也看不懂）**。

### 一、 访问控制：IAM Policy vs. Bucket Policy

在 AWS 中，决定“某人/某组件能否读写某个 S3 桶”主要靠两种策略（Policy）。它们都是 JSON 格式，遵循显式拒绝优先（Explicit Deny overrides Allow）的原则。

#### 1\. IAM Policy（基于身份的策略）

- **定义**：挂载在 **IAM 用户（User）、用户组（Group）或 IAM 角色（Role）** 上的策略。
- **视角**：从“人或组件”的角度出发——**“我作为一个 Lambda 函数，我能对哪些 S3 桶做什么？”**
- **适用场景**：当你需要管理某个特定程序（如 Airflow 调度器、EC2 实例上的数据抽取脚本）的权限时使用。

#### 2\. Bucket Policy（基于资源的策略）

- **定义**：直接挂载在 **S3 Bucket 本身** 上的策略。
- **视角**：从“存储桶”的角度出发——**“我作为一个桶，我允许哪些人、来自哪个 AWS 账号的请求来读写我？”**
- **适用场景**：
	- **跨账号访问（Cross-Account Access）**：允许公司另一个 AWS 账号下的数据分析师读取当前账号的数据湖。
		- **强制安全合规**：例如，规定“任何上传到这个桶的对象，必须带有加密属性，否则直接拒绝”。


```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowCrossAccountRead",
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::123456789012:root"
            },
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-data-lake-bucket",
                "arn:aws:s3:::my-data-lake-bucket/*"
            ]
        }
    ]
}
```

> ⚠️ **新手踩坑点**：在 `Resource` 中，`arn:aws:s3:::bucket-name`（不带斜杠）代表**存储桶本身**（用于 `ListBucket` 列出文件列表）；而 `arn:aws:s3:::bucket-name/*`（带斜杠通配符）代表**桶里的对象**（用于 `GetObject` 或 `PutObject` 读写文件）。两者必须协同配置！

### 二、 网络防线：阻止公共访问与 VPC 终端节点

### 1\. Block Public Access (BPA) —— 绝对防火墙

早期许多开发者误将 Bucket Policy 设为公开，导致数据泄露。现在 AWS 引入了 **Block Public Access** 托管开关（默认全开）。

- 它是覆盖在桶或整个账号最外层的“紧箍咒”。
- 只要开启了 BPA，哪怕你的 Bucket Policy 写了 `Effect: Allow, Principal: *`（允许所有人），S3 也会**强行拦截所有来自公网的匿名访问**。在生产环境中，除了存放前端静态网站的桶，其余所有数据管道的桶都必须开启 BPA。

### 2\. VPC Endpoint（VPC 终端节点）—— 内网管道

如果你的数据清洗程序运行在私有子网的 EC2、Glue 或本地机房里，它们没有外网 IP，该如何访问 S3？

- **走公网（错误做法）**：让私有流量通过 NAT 网关路由到外网，再访问 S3 的公网 API。这会产生高昂的 NAT 流量费，且不安全。
- **VPC Endpoint（正确做法）**：在 VPC 内创建一个 **Gateway Endpoint**。它会在你的路由表里加一条记录，让原本要去往 S3 的流量**完全在 AWS 内部骨干网络中传输**，绝不流向公网，既安全又免费。

### 三、 数据加密（Data Encryption）

即使有人物理盗取了 AWS 塞满硬盘的服务器，或者通过漏洞拿到了底层数据，加密也能确保他们看到的是乱码。

#### 1\. 静态加密 (Encryption at Rest)

目前 S3 已经实现**默认全员静态加密**（采用 AES-256 算法）。主要有两种主流管理方式：

- **SSE-S3 (Amazon S3 托管密钥)**：完全托管。AWS 自动帮你处理密钥的创建、轮换和管理，你不需要额外付钱，也不用写复杂的代码。适合绝大多数通用场景。
- **SSE-KMS (AWS Key Management Service 密钥)**：企业级控权。你可以自己控制密钥的轮换策略、审计日志（谁在什么时间调用了解密密钥）。
	- *数据工程考量*：使用 KMS 加密后，当 Lambda 想要从 S3 读文件时，它不仅需要 S3 的 `GetObject` 权限，还需要 KMS 的 `kms:Decrypt` 权限，权限控制更加精细。

#### 2\. 传输中加密 (Encryption in Transit)

为了防止数据在网络传输过程中被“中间人攻击”窃听，必须强制使用 **HTTPS**。 可以通过 Bucket Policy 增加一条显式的 `Deny` 规则：如果请求的 `aws:SecureTransport` 为 `false`（即使用的是不安全的 HTTP），则直接拒绝访问。

### 四、 权限排查的“黄金法则”

当你在跑数据管道（如 Airflow 往 S3 推送数据，或者 Lambda 读取 S3 触发器文件）遇到 `403 Access Denied` 时，请按照以下顺序“连问四题”排查：

1. **IAM 角色对了吗？** 运行该代码的组件（Lambda/EC2）是否挂载了包含 `s3:PutObject` 或 `s3:GetObject` 权限的 IAM 角色？
2. **Bucket Policy 挡了吗？** 目标桶的 Bucket Policy 是否有显式的 `Deny` 语句误伤了当前身份？或者是跨账号访问时忘记在 Principal 里加白名单？
3. **KMS 权限给了吗？** 如果桶使用了自定义的 KMS 密钥加密，你的 IAM 角色除了 S3 权限，有没有被授予该 KMS 密钥的 `Decrypt/GenerateDataKey` 权限？
4. **网络通不通？** 处于私有子网的组件是否配置了 VPC Endpoint 从而能够直达 S3 的网络边界？
