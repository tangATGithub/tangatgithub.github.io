# S3
## 1. 核心概念：它不是传统文件系统

S3 是**对象存储（Object Storage）**，而不是分层文件系统。

- **Bucket（存储桶）**：存储对象的容器。**Bucket 名称在全球所有 AWS 账户中必须是唯一的**。
- **Object（对象）**：你上传的文件。每个对象由两部分组成：
	- **Data（数据本身）**
	- **Metadata（元数据）**：键值对（Key-Value pairs），记录文件的类型、大小、创建时间或自定义标签。
- **Key（键）**：文件的“绝对路径”（例如 `raw-data/2026/06/09/log.json`）。在底层，S3 是扁平的键值结构，UI 里的“文件夹”其实只是由分隔符（`/`）产生的前缀（Prefix）视觉效果。
- **无限容量与单文件上限：** 你可以在一个 S3 桶（Bucket）里塞进任意 PB 级别的数据，它会自动横向扩展。但注意：**单个 Object（文件）的最大上限是 5 TB**。如果超过 5 TB，必须使用分段上传（Multipart Upload）。


- **Durability（持久性 - 11个9）**：`99.999999999%`。这意味着如果你在 S3 中存储 10,000 个对象，平均每 1000 万年才会丢失一个文件。S3 默认会把你的文件自动复制到同一 Region 内的至少 3 个物理可用区（AZ）。
- **Availability（可用性 - 4个9）**：`99.99%`。指服务在一年中可以正常访问的时间比例。S3 standard has 99.99% availability = not available 53 min a year.


## 2. 版本控制 versioning

Amazon S3 的 **Versioning（版本控制）** 是一个在**存储桶（Bucket）级别**开启的功能。简单来说，它就像是为你的 S3 存储桶配备了一个“时光机”**和**“回收站”。

在理解 Versioning 时，最核心的观念是：**开启版本控制后，S3 桶中的文件永远不会被真正“覆盖”或“直接删除”，而是会保留所有的历史痕迹。**

### 1\. 核心工作原理

#### A. 文件的“覆盖”操作 

- **未开启 Versioning**：如果你上传了一个名为 `config.json` 的文件，接着又上传了一个同名的 `config.json`，旧文件的内容会被新内容**直接覆盖**，旧数据永久丢失。
- **开启 Versioning**：每次你上传同名文件时，S3 不会覆盖原文件，而是会为这个文件生成一个唯一的 **Version ID**（版本 ID）。
	- 最新的那次上传会被标记为 **Current Version（当前版本）**。
		- 之前上传的同名文件会变成 **Noncurrent Version（历史版本）**。
		- 当你通过代码或 API 请求 `config.json` 且不指定版本号时，S3 默认永远返回 **Current Version**。

#### B. 文件的“删除”操作 


在开启版本控制的桶里，当你点击“删除”一个文件时：

1. S3 并没有真正抹去任何数据。
2. S3 会在最上面放置一个特殊的对象，叫做 **Delete Marker（删除标记）**。
3. 这个 **Delete Marker** 顶替它成为了 **Current Version（当前版本）**。
4. 此时你在 AWS 控制台（默认视图）或通过标准 API 访问该文件，S3 会返回 `404 Not Found`，假装文件已经被删除了。

> 💡 **如何“复活”被删的文件？** 你只需要在控制台打开“显示版本（Show versions）”开关，找到那个处于最顶层的 `Delete Marker` 并把它**彻底删除**。一旦删除了删除标记，前一个历史版本就会重新变成“当前版本”，文件完美复活。

---
### 2\. 状态切换与收费逻辑

关于 Versioning 的状态，你需要知道以下三点：

1. **状态不可逆（只能开启或暂停）**： 一个 S3 桶的 Versioning 状态只有三种：
	- 未开启（Unversioned）
	- 已开启（Enabled） 
	- 已暂停（Suspended）。一旦开启过，你就再也无法把它变回“未开启”状态，只能选择“暂停”。
2. **暂停后的行为**： 如果暂停了版本控制，新上传的同名文件会覆盖 `Version ID` 为 `null` 的那个版本，但**之前已经存在的历史版本依然会安全地躺在里面**，不会消失。
3. **空间与计费**： **每一个历史版本都会按照实际大小完整计费！** 比如一个 10GB 的文件，你修改并上传了 5 次，在开启版本控制的情况下，S3 里会堆积 5 个版本，总共计算 **50GB** 的存储费用。这也是为什么不能盲目开启版本控制、或者==开启后必须配合生命周期管理的原因==。

### 3\. 彻底删除（Permanent Delete）怎么做？

如果你因为合规、清空空间等原因，必须**彻底、物理删除**一个文件，你需要怎么做？

- 你必须在删除请求中**明确指定 `Version ID`**。
- 只有显式指定了版本 ID 的删除操作，S3 才会把该版本从硬盘上永久抹去。
- 如果你要清空桶，必须写脚本或配置规则，把所有的历史版本和所有的 `Delete Marker` 全部彻底删除。

### 4\. 数据工程中的生产实践

在实际的数据管道（ETL）和数据湖架构中，Versioning 通常与以下特性结合使用：

- **防止误操作/勒索病毒（MFA Delete）**： 可以配置为：如果想要“彻底删除”某个历史版本或关闭版本控制，必须提供根账号的 **MFA（多因素验证码）**。这样即使生产环境的 IAM 密钥泄露，黑客也无法恶意清空你的历史数据资产。
- **配合生命周期策略（Lifecycle Rules）进行成本控制**： 因为历史版本很费钱，通常会配置生命周期规则：
	> “当前版本（Current）正常保留；一旦变成历史版本（Noncurrent），30天后自动转入便宜的 `Glacier` 归档存储，90天后彻底物理删除。”
- **数据审计与回滚**： 在机器学习训练或数据分析中，如果你发现今天的模型训练结果异常，怀疑是输入的数据源被污染了，你可以通过 Versioning 轻松追溯并读取到昨天、甚至前天那个时间节点的历史数据快照。



## S3 replication
**Amazon S3 Replication** 是 AWS 提供的一项功能，用于在不同的 Amazon S3 存储桶（Buckets）之间实现**自动、异步的跨区域或同区域对象复制**。

只要你在源存储桶上配置了复制规则，任何**新上传**的对象、对象更新或标签更改，都会被 S3 自动且在后台复制到目标存储桶中。

### 核心工作原理

S3 Replication 核心要求源存储桶和目标存储桶都**必须开启版本控制（Versioning）**。它主要分为两种类型：

- **CRR (Cross-Region Replication，跨区域复制)：** 源存储桶和目标存储桶位于**不同的 AWS 区域**（例如：从美西旧金山复制到美东弗吉尼亚）。
- **SRR (Same-Region Replication，同区域复制)：** 源存储桶和目标存储桶位于**相同的 AWS 区域**。

### 典型使用场景

以下是几种最常见的工业级应用场景：

#### 1\. 合规性与容灾备份 (DR)

- **多区域容灾：** 为了防止某个 AWS 区域发生罕见的重大灾难或基础设施故障，使用 **CRR** 将核心数据实时备份到另一个地理位置相隔很远的区域，确保业务具备超强的容灾恢复能力（DR）。
- **主备切换：** 当主业务区不可用时，应用可以迅速切换到备用区域的 S3 存储桶读取数据。

#### 2\. 降低全球用户的访问延迟 (Latency Optimization)

- 如果你的公司总部在美国，但亚洲有大量的用户。直接让亚洲用户跨洋访问美国的 S3 存储桶速度会很慢。
- 通过 **CRR** 将数据复制到东京或新加坡的存储桶，让亚洲的用户直接访问本地的存储桶，大幅提升下载和加载速度。

#### 3\. 数据主权与合规性要求 (Compliance)

- 某些国家的法律法规（如 GDPR 或特定金融监管）严格要求某些特定数据必须存储在境内。
- 你可以通过复制规则，将特定前缀（Prefix）或带有特定标签（Tags）的数据自动路由并存储到符合合规要求的特定区域存储桶中。

#### 4\. 生产与测试环境的数据隔离 (Environment Isolation)

- 在日常开发中，可能需要使用生产环境的真实数据来做大数据分析、机器学习模型训练或严格的测试。
- 为了防止分析或测试脚本不小心误删、污染生产数据，可以使用 **SRR** 或 **CRR** 将生产存储桶的数据自动同步到一个完全独立的“分析/测试存储桶”中，实现安全隔离。

#### 5\. 跨账号所有权转移 (Cross-Account Ownership)

- 在企业多账号管理架构下（比如一个专门的 Log 账号，一个 Production 账号），可能需要将数据从一个 AWS 账号迁移或分享给另一个 AWS 账号。
- S3 Replication 支持跨账号复制，并且可以在复制的同时**自动更改目标对象的所有权（Ownership）**，方便接收方账号直接控制和管理这些文件。

### 关键技术特性（常考）

- **异步复制：** 复制不是瞬间完成的（非同步写入）。虽然大部分对象在几分钟内就能完成复制，但标准服务并不打包票具体的完成时间。如果对时间有严格要求，可以开启 **S3 Replication Time Control (RTC)**，它承诺 99.99%的对象会在 15 分钟内复制完成。
- **非追溯性（历史数据默认不复制）：** 在你开启复制规则的那一刻起，只有**新上传**或**被修改**的对象会被自动复制。如果你需要复制开启前存储桶里的“存量历史数据”，需要配合使用 **S3 Batch Operations（批处理操作）** 来完成。
- **删除行为（Delete Markers）：** 默认情况下，如果你在源存储桶删除了一个文件（产生了一个删除标记），这个删除标记**不会**复制到目标存储桶，这样可以防止源存储桶被恶意投毒或误删时，备份数据也被瞬间毁掉。当然，这个行为可以通过配置来调整。

## S3 Storage Class（存储类）


企业里每天会产生海量的数据。如果把“3年前基本没人看的历史老账本”和“当前每秒都在写入的 Kinesis 实时天气流”放在同一种高配磁盘里，公司的财务主管会直接疯掉。

为此，S3 提供了 6 个不同的**存储类（分层）**，核心逻辑就是：**用“数据访问的频次”来置换“存储成本”**。注意：storage class可以是object level，不一定是bucket level。

S3 的计费主要挂钩于**存储容量**和**访问/检索频率**。合理使用存储类能帮你省下超过 50% 的账单：

| 存储类 (Storage Class)                       | 适用场景                                 | 场景举例                                                 | 特点                                                                                                         |
| ----------------------------------------- | ------------------------------------ | ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| S3 Standard (标准)                      | 频繁访问的数据（如热数据、API 响应、网站静态资源）          | Kinesis 正在实时落地的原始 CSV 文件夹，或者分析师每天都要高频 `SELECT` 的热数据。 | 低延迟、高吞吐，存储费最贵，但读取（Retrieval）不要钱                                                                            |
| S3 Standard-IA (不频繁访问)                | 数据不常用，但一旦要查，必须毫秒级立刻弹出来。          | 比如公司上个月的天气报表。平时没人看，但突然遇到紧急审计或复查时，必须马上能点开。            | 存储费大约比 Standard 便宜 40%，但是每次你去读取（下载/查询）数据时，AWS 要额外收取一笔“数据提取费”。                                              |
| S3 One Zone-IA                        | 容易重建的不频繁访问数据（如临时生成的缩略图、非关键缓存）        | 比如你用 Athena 转换出来的 Parquet 备份表，哪怕丢了，大不了用原始 CSV 再转一次。  | 只存放在 1 个可用区，成本比 Standard-IA 再低 20%，但有丢失风险。                                                                 |
| S3 Glacier Instant Retrieval（冰川-即时检索）<br> | 季度级别（每季度查个一两次）的归档数据，但保留毫秒级的响应速度。 | 全年的历史消费底单备份。                                         | 存储费极低，提取费比 IA 更高。                                                                                          |
| S3 Glacier Flexible Archive           | 归档、合规备份数据（几年不看一次）                    | 公司的合规性历史审计日志，可能一年才看一次。                               | 当你发起查询申请后，数据不能立马出来。==根据你付的加急费==(flexible的来源)，需要等待 1分钟到5小时（AWS 会在后台把数据从磁带机或冷冷盘里“解冻”到标准层，你才能用 Athena 查）。 |
| S3 Glacier deep Archive               | 极其缓慢，解冻需要 12 到 48 小时。            | 医疗影像备份、或者是法律规定必须留存 7 年以上的死数据。                        | 存储费极其夸张的便宜（大约是 Standard 的九分之一）。                                                                            |
| S3 Intelligent-Tiering                | 访问模式未知或多变的数据（大数据管道、数据湖首选）        |                                                      | 自动监控访问模式，在频繁/不频繁层之间切换，无检索手续费。                                                                          |
