# S3
## 1. 核心概念：它不是传统文件系统

S3 是**对象存储（Object Storage）**，而不是分层文件系统。

- **Bucket（存储桶）**：存储对象的容器。**Bucket 名称在全球所有 AWS 账户中必须是唯一的**。
- **Object（对象）**：你上传的文件。每个对象由两部分组成：
	- **Data（数据本身）**
	- **Metadata（元数据）**：键值对（Key-Value pairs），记录文件的类型、大小、创建时间或自定义标签。
- **Key（键）**：文件的“绝对路径”（例如 `raw-data/2026/06/09/log.json`）。在底层，S3 是扁平的键值结构，UI 里的“文件夹”其实只是由分隔符（`/`）产生的前缀（Prefix）视觉效果。
- **无限容量与单文件上限：** 你可以在一个 S3 桶（Bucket）里塞进任意 PB 级别的数据，它会自动横向扩展。但注意：**单个 Object（文件）的最大上限是 5 TB**。如果超过 5 TB，必须使用分段上传（Multipart Upload）。


## 2. 版本控制 versioning

Amazon S3 的 **Versioning（版本控制）** 是一个在**存储桶（Bucket）级别**开启的功能。简单来说，它就像是为你的 S3 存储桶配备了一个“时光机”**和**“回收站”。

在理解 Versioning 时，最核心的观念是：**开启版本控制后，S3 桶中的文件永远不会被真正“覆盖”或“直接删除”，而是会保留所有的历史痕迹。**

### 核心工作原理

#### 文件的“覆盖”操作

- **未开启 Versioning**：如果你上传了一个名为 `config.json` 的文件，接着又上传了一个同名的 `config.json`，旧文件的内容会被新内容**直接覆盖**，旧数据永久丢失。
- **开启 Versioning**：每次你上传同名文件时，S3 不会覆盖原文件，而是会为这个文件生成一个唯一的 **Version ID**（版本 ID）。
	- 最新的那次上传会被标记为 **Current Version（当前版本）**。
		- 之前上传的同名文件会变成 **Noncurrent Version（历史版本）**。
		- 当你通过代码或 API 请求 `config.json` 且不指定版本号时，S3 默认永远返回 **Current Version**。

#### 文件的“删除”操作 


这是最容易让人产生误解的地方。在开启版本控制的桶里，当你点击“删除”一个文件时：

1. S3 并没有真正抹去任何数据。
2. S3 会在最上面放置一个特殊的对象，叫做 **Delete Marker（删除标记）**。
3. 这个 **Delete Marker** 顶替它成为了 **Current Version（当前版本）**。
4. 此时你在 AWS 控制台（默认视图）或通过标准 API 访问该文件，S3 会返回 `404 Not Found`，假装文件已经被删除了。

> 💡 **如何“复活”被删的文件？** 你只需要在控制台打开“显示版本（Show versions）”开关，找到那个处于最顶层的 `Delete Marker` 并把它**彻底删除**。一旦删除了删除标记，前一个历史版本就会重新变成“当前版本”，文件完美复活。

### 状态切换与收费逻辑

关于 Versioning 的状态，你需要知道以下三点：

1. **状态不可逆（只能开启或暂停）**： 一个 S3 桶的 Versioning 状态只有三种：
	- 未开启（Unversioned）
	- 已开启（Enabled） 
	- 已暂停（Suspended）。一旦开启过，你就再也无法把它变回“未开启”状态，只能选择“暂停”。
2. **暂停后的行为**： 如果暂停了版本控制，新上传的同名文件会覆盖 `Version ID` 为 `null` 的那个版本，但**之前已经存在的历史版本依然会安全地躺在里面**，不会消失。
3. **空间与计费**： **每一个历史版本都会按照实际大小完整计费！** 比如一个 10GB 的文件，你修改并上传了 5 次，在开启版本控制的情况下，S3 里会堆积 5 个版本，总共计算 **50GB** 的存储费用。这也是为什么不能盲目开启版本控制、或者==开启后必须配合生命周期管理的原因==。

### 数据工程中的生产实践

在实际的数据管道（ETL）和数据湖架构中，Versioning 通常与以下特性结合使用：

- **防止误操作/勒索病毒（MFA Delete）**： 可以配置为：如果想要“彻底删除”某个历史版本或关闭版本控制，必须提供根账号的 **MFA（多因素验证码）**。这样即使生产环境的 IAM 密钥泄露，黑客也无法恶意清空你的历史数据资产。
- **配合生命周期策略（Lifecycle Rules）进行成本控制**： 因为历史版本很费钱，通常会配置生命周期规则：
	> “当前版本（Current）正常保留；一旦变成历史版本（Noncurrent），30天后自动转入便宜的 `Glacier` 归档存储，90天后彻底物理删除。”
- **数据审计与回滚**： 在机器学习训练或数据分析中，如果你发现今天的模型训练结果异常，怀疑是输入的数据源被污染了，你可以通过 Versioning 轻松追溯并读取到昨天、甚至前天那个时间节点的历史数据快照。
